# Skill Authoring Standard

Use this when creating or updating skills so they stay clear, reusable, and easy to trust.

## Goal

A skill should be easy to understand at a glance:
- what it is for
- when to use it
- what it assumes
- what to avoid

## Default Skill Structure

Use these sections when they fit:

### Title
A clear name.

### Goal
One short statement of the problem the skill solves.

### When to use
Describe the situations that should trigger this skill.

### Prerequisites
List required tools, files, access, context, or assumptions.
If there are none, omit the section.

### Workflow or rules
The main behavior, steps, or decision rules.
Keep it practical.

### Do
Short list of habits that help.

### Don't
Short list of mistakes to avoid.

### Output style
If the skill affects how results should be presented, say so.

### Notes
Only include durable details worth keeping.

## Writing Rules

- Prefer plain language over framework language.
- Keep the behavior, discard the branding.
- Do not pad with sections that add nothing.
- Do not turn a small habit into a manifesto.
- If a skill is project-specific, say so plainly.
- If a skill depends on certain tools, state that up front.

## Good Skill Traits

A good skill is:
- reusable
- specific enough to change behavior
- short enough to scan
- durable across sessions

## Bad Skill Traits

Avoid skills that are:
- vague
- mostly identity fluff
- copied product docs with names swapped out
- dependent on infrastructure the assistant does not actually have
- duplicating another skill without adding anything

## Notes

The point of a skill is reliable reuse, not impressive formatting.
